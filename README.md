# Promo Service

Сервис для создания заказов с поддержкой промокодов. Реализован как тестовое `prod-like` задание в компанию JetLend.

Принимает заказ от пользователя, валидирует промокод по набору бизнес-правил и рассчитывает итоговую стоимость с учётом скидки. Промокод может быть ограничен по категории товаров, количеству использований и сроку действия - при этом отдельные товары можно исключить из акций.

## Quick start

### Docker (рекомендуется)

Клонируем репозиторий и создаём `.env` из примера:

```bash
git clone https://github.com/Dezmound1/PromoService.git
cd PromoService
cp env.example .env
```
В `.env` выставляем `IS_LOCAL=0` ОБЯЗАТЕЛЬНО! - сервис переключится на Postgres:

Поднимаем всё командами:

```bash
make build
make run
```

Сервис будет доступен на `http://localhost:8000`, Swagger - на `http://localhost:8000/api_v1/`.

Остановить:

```bash
make stop
```

### Локально (без Docker)

Если хотите запустить без контейнеров - понадобится [uv](https://docs.astral.sh/uv/) и Python 3.10+.

```bash
uv sync
```

В `.env` выставляем `IS_LOCAL=1` - сервис переключится на SQLite:

```bash
uv run promo_service/manage.py migrate
uv run promo_service/manage.py seed_db
uv run promo_service/manage.py runserver
```

`seed_db` наполнит базу тестовыми данными (пользователи, товары, промокоды). Флаг `--clear` очистит таблицы перед заполнением.


## Задание

Реализовать HTTP endpoint для создания заказа. При создании заказа пользователь может указать промокод, который даёт процентную скидку (А может и не указывать).

### Правила применения промокода

- Промокод должен существовать
- Промокод не должен быть просрочен
- У промокода есть ограничение на максимальное количество использований
- Один пользователь не может использовать один и тот же промокод более одного раза
- Промокод может быть ограничен определённой категорией товаров
- Некоторые товары могут быть исключены из любых акций, к ним промокод не должен применяться

### Формат запроса

`POST /orders/`

```json
{
  "user_id": 1,
  "goods": [
    { "good_id": 1, "quantity": 2 }
  ],
  "promo_code": "SUMMER2025"
}
```

### Формат ответа

```json
{
  "user_id": 1,
  "order_id": 1,
  "goods": [
    {
      "good_id": 1,
      "quantity": 2,
      "price": 200,
      "discount": "0.1",
      "total": 180
    }
  ],
  "price": 200,
  "discount": "0.1",
  "total": 180
}
```

Где 
`price` - сумма без скидки, 
`discount` - доля скидки, 
`total` - итого к оплате. 

Если промокод не применим к конкретному товару (категория не совпадает или товар исключён из акций), скидка на него не распространяется.

## Выбор стратегии

### Архитектура

Проект разбит на три слоя:

- **Repository** - работа с БД. Никакой логики, только запросы: создать, найти, посчитать.
- **Service** - бизнес-логика. Валидация промокода, расчёт скидок, сборка заказа.
- **View** - принимает запрос, вызывает сервис, возвращает ответ.

Основная логика живёт в двух сервисах:
- `PromoCodeService` - валидация промокода по правилам из задания
- `OrderService` - в сервисе заказов происходит вызов валидации промокода, подсчет цены заказа и каждого товара с учетом (или без) скидки, создаёт заказ

### Decimal

При работе с вычислительными операциями расчёта скидки был использован `Decimal` для точных вычислений. ```Ходит байка в интернетах, что это так называемый "best practice"``` - ну а если серьёзно, `float` теряет точность на деньгах, и это реальная проблема.

### Тесты

Использую `pytest` + `pytest-django` - просто привык к нему, нравится синтаксис фикстур и параметризация. Покрытие проверяется через `pytest-cov`, порог 80% в pre-commit. Коммит не пройдёт, если покрытие меньше указанного порога.

### Docstrings

Docstrings написаны в формате (`Parameters`, `Returns` с разделителями `----------`). Такой стиль принят в текущей компании, где я работаю - решил придерживаться его и здесь для единообразия.
